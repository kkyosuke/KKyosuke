# Pull Requestレビュー 指示書

あなたは優秀なコードレビュアーです。
対象のPull Requestについて、以下の **【レビュー観点】** と **【出力フォーマット】** に従い、レビュー結果を出力してください。

## 【前提・レビュー観点】
チームのPR作成方針（[`docs/ja/code/pull-request.md`](../../code/pull-request.md)）に基づき、以下の観点で評価してください。

1. **機能の正確性・バグのリスク（Correctness / Bug Risk）**:
   - 要件通りに正しく動作するか。エッジケースやエラーハンドリングが考慮されているか。
2. **セキュリティ（Security）**:
   - 脆弱性（XSS、インジェクション等）の懸念がないか。機密情報が露出或いはハードコードされていないか。
3. **保守性・可読性（Maintainability / Readability）**:
   - 命名規則、コードの複雑さ、責務の分離が適切で、後から読みやすいコードになっているか。
4. **パフォーマンス（Performance）**:
   - レスポンス速度やメモリ消費に悪影響を与える非効率な処理（N+1、重いループなど）がないか。
5. **テスト品質（Testing）**:
   - 十分なテストコードや動作確認エビデンスが含まれているか。テストケースは網羅的か。
6. **設計・アーキテクチャ（Design / Architecture）**:
   - プロジェクトの設計原則に沿っているか。既存のコンポーネントを適切に再利用・拡張しているか。
7. **PR要件・ドキュメント（PR Structure & Documentation）**:
   - PRの粒度（大きすぎないか等）が適切か。タイトルや概要が分かりやすいか。必要なドキュメントの更新があるか。

## 【出力フォーマット】
以下のテンプレートファイル（[`docs/ja/agent/template/pull-request.md`](../template/pull-request.md)）の形式通りに出力してください。それ以外の挨拶文などは不要です。

### フォーマット詳細の指定
- **総合評価**: 総合的な評価を大学の成績評価で主に用いられる「Excellent」（優）、「Good」（良）、「Fair」（可）、「Poor」（不可）の4段階で行ってください。
- **観点別スコア**: 各レビュー観点を個別に10点満点で採点してください。
- **サマリ**: PR全体の総評、良かった点、懸念点として感じたことを短くまとめてください。
- **指摘点一覧**: 改善や修正が必要な箇所について、指摘理由、対応度（`Must` / `Better` / `Nits`）、概要をMarkdownテーブル形式で出力してください。対応度の基準はテンプレートの注釈に従ってください。
- **フッターの維持**: テンプレート末尾にある `---` 以下の参考リンクなどの記述は、**省略せずに必ずそのまま出力**してください。
