freee 連携 (OAuth 2.0) 認証フロー

このドキュメントでは、Slackアプリからfreee人事労務へ連携（OAuth 2.0 認可コードフロー）する際の一連のアーキテクチャとシーケンスを解説します。

概要

Slack Block Kitの仕様上、バックエンド側から強制的にユーザーのブラウザを開かせることはできません。そのため、ボタン要素に url プロパティを付与し、Slackクライアントにブラウザ起動を委譲しています。
また、ユーザー体験（UX）を向上させるために、即座にfreeeの認可画面へリダイレクトさせず、一度「ローディング画面（クッションページ）」を挟むアプローチを採用しています。

各エンドポイントの役割

1. GET /freee/auth/start

  • 役割: クッションページ（ローディング画面）の表示。
  • 背景: ボタンクリック直後にいきなりfreeeの画面に飛んだり、APIの応答待ちでブラウザが真っ白になるのを防ぐため、明示的に「遷移中」であることをユーザーに知らせます。
  • 処理: CSSによるスピナーと遷移メッセージを含むHTMLを返し、1秒後に /freee/auth へJSでリダイレクトします。

2. GET /freee/auth

  • 役割: 認可URLの生成とリダイレクト。
  • 処理:
    1. クエリパラメータから user_id を取得。
    2. CSRF攻撃を防ぐための state パラメータを生成（userId + UUID）。
    3. state をセキュアなCookie（freee_auth_state）に保存。
    4. freeeクライアントで認可URL（/public_api/authorize）を生成し、302リダイレクト。

3. GET /freee/auth/callback

  • 役割: 認可コードの受け取りとアクセストークン取得、DB保存、Slack UI更新。
  • 処理:
    1. 送られてきた state がCookieの値と一致するか検証（CSRF対策）。
    2. state に埋め込まれた user_id を抽出。
    3. 認可コード (code) を用いて、freee APIにトークンリクエストを実行。
    4. 取得した access_token および refresh_token を user_tokens テーブルへ保存（または更新）。
    5. 連携が完了したことを反映するため、Slack API (views.publish) を呼び出してユーザーのホームタブを更新（未出勤状態などにする）。
    6. タブを自動で閉じる（window.close()）スクリプトを含む認証完了HTMLを返却。

データベーススキーマについて

アクセストークンおよびリフレッシュトークンは、user_tokens テーブルで管理されます。

カラム名 説明
id UUID (主キー)
user_id SlackのユーザーID (Uxxxxxxxx)
service 連携サービス名 (freee など)
type トークンの種類 (access_token または refresh_token)
token トークン文字列
expires_at トークンの有効期限 (ISO8601形式文字列)

※ (user_id, service, type) の組み合わせでユニーク制約が設定されています。
