GitHub App Webhook Event Routing

このドキュメントでは、src/handlers/webhook.ts に実装されている GitHub Webhook イベントのルーティングロジックについてまとめています。

全体フロー図

各イベントを受信した際の、大まかな条件分岐と呼び出されるジョブ（コマンド）のフローです。

ルーティング詳細

それぞれのイベントにおける詳細な発火条件と、実行される内容です。

イベント アクション 条件 ルーティング先 (処理内容)
issue_comment edited Pull Requestへのコメント
＋ 編集により再レビューチェックボックスがONになった		 reReviewCommand
再レビュー処理を開始します。
created Pull Requestへの新規コメント routeCommentCommand
コメント内容に応じたルーティング処理へ委譲します。
pull_request_review_comment created ボット以外のコメント
＋ (メンションが含まれる または 既存スレッドへの返信)		 replyCommand
コード行に対するレビューコメントへの応答処理を行います。
pull_request synchronize Pull Requestへの新規コミットのプッシュ等 Cloudflare KV 操作
実行中の古いレビュー処理を停止するためのキャンセルシグナルフラグ（TTL付き）を保存します。
補足事項（バックグラウンド実行）
ルーティング先の各コマンド（reReviewCommand, routeCommentCommand, replyCommand）は、Cloudflare Workers 等の環境では c.executionCtx.waitUntil() を使用してバックグラウンドタスクとして非同期に実行されます。
Webhook 自体のリクエストに対しては即座に HTTP 200 OK を返し、GitHub 側でのタイムアウトを防ぐ設計になっています。
