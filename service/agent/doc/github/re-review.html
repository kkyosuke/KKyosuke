再レビュー時の未解決チェック設計 (GitHub機能依存)

このドキュメントでは、コードの再レビュー (Re-Review) を行う際に、GitHubのネイティブなスレッド管理機能（GraphQL API等）を用いて効率的かつ正確にスレッドの解決状態を判定する仕組みについてまとめています。

1. アーキテクチャ方針

複数回のレビューサイクルが回った場合でも、過去の指摘を含めて「AIボットが指摘した項目がすべて解決されているか」を担保する必要があります。

解決策: 独自のID管理を行わず、GitHubのGraphQL APIを使用してPull Request上のすべてのReview Threadを取得します。その中から「AIボットが作成したスレッド」を抽出し、そのうちの1つでも isResolved: false の状態であれば、再レビューをブロックして修正を促すようにします。

3. スレッド状態の取得と判定

再レビューのトリガー後、GraphQLの pullRequest.reviewThreads を用いてデータを取得し、以下のように判定します。

// GitHub GraphQL API で全スレッドを取得済みとする (reviewThreads) const unresolvedBotThreads = (reviewThreads || []).filter(thread => { // すでに解決済み、あるいはコメントがないスレッドはスキップ if (thread.isResolved || !thread.comments?.nodes?.length) return false; // スレッドの最初のコメントの作成者を取得 const firstCommentAuthor = thread.comments.nodes[0].author?.login?.toLowerCase() || ""; // 作成者がボット(AI)である未解決スレッドのみを抽出 return firstCommentAuthor.includes("bot") || firstCommentAuthor.includes("ai"); }); const hasUnresolvedBotThreads = unresolvedBotThreads.length > 0;
実装のポイント (src/jobs/re-review/agent.ts)
  • hasUnresolvedBotThreads が true の場合、LLMを利用した全体の再レビュー処理（コスト大）をスキップします。
  • 「未解決のコメントが残っています」という警告コメントを投稿し、開発者にResolveを促します。
  • 過去のどのレビューサイクルの指摘であっても、未解決である限りは再レビューを行わず、漏れを防ぐことができます。
