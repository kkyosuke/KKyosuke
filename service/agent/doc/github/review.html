初回レビューの基本設計 (GitHub機能への集約)

このドキュメントでは、コードレビュー開始時にGitHubの標準機能を活用して状態を管理する設計についてまとめています。独自のID（Review ID）や外部のKVストアは使用せず、GitHub APIが提供する「Review」と「Thread」の概念に完全に依存します。

1. 目的とアーキテクチャ方針

  • 非同期処理アーキテクチャ (Cloudflare Queues): LLMを用いたコードレビューは30秒以上かかることが多く、Cloudflare Workersの実行時間制限（HTTP Fetch時の30秒）を回避するため、Webhook受領時はキューへのメッセージ送信（Publish）のみ行い、実際の処理はキューのConsumer（最大15分実行可能）で非同期に行います。
  • ステートレス化: KVストア等での独自の状態管理を廃止し、GitHub自体を唯一のデータソース（Single Source of Truth）とします。
  • スレッド機能の活用: GitHubのインラインコメントは自動的に「スレッド」を形成し、isResolved（解決済みかどうか）のフラグを持ちます。これを利用することで、未解決指摘のトラッキングをAPIのみで実現します。

3. 実装のポイント

実装箇所 (src/handlers/webhook.ts & src/jobs/github/queue.ts & src/jobs/github/review/agent.ts)
  • Webhook Handler: GitHub Webhookを受け取ると、処理に必要なペイロード（PR情報等）を ai-kyosuke-queue にPublishし、即座に200 OKを返します。
  • Queue Consumer: バックグラウンドでQueueメッセージを受信し、generateCodeReview でLLMからレビュー結果を取得します。
  • createReview APIを呼び出し、Pull Requestのメインレビュー（全体のサマリと評価）を投稿します。
  • createReviewComment APIを使用して、具体的なコード行に対してインラインコメントを投稿します。この際、GitHub側で自動的に一意な「スレッドID」が生成され、未解決 (Unresolved) 状態でトラッキングが開始されます。
