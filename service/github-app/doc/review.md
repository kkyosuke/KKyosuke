# PR Review Agent 設計ドキュメント

## 概要
GitHub の Webhook をフックとし、Pull Request のコメントに特定のキーワードが含まれていた場合に、LLM を用いた自動コードレビューを実行するエージェントの設計です。

## ワークフローとアーキテクチャ

### 1. Webhook の受信とイベント検証
- **エンドポイント:** Hono サーバー上に Webhook 受信用エンドポイント（例: `POST /api/webhook/github`）を構築します。
- **セキュリティ:** GitHub からのシークレットキーを用いて、Webhook ペイロードの署名検証を行い、正当なリクエストであることを確認します（`@octokit/webhooks` の利用を推奨）。

### 2. トリガーのフィルタリング（発火条件）
以下の条件をすべて満たした場合にのみ、Agent を稼働させます。
- **イベントタイプ:** `issue_comment` の `created` アクションであること。
- **PR 判定:** Payload 内に `issue.pull_request` オブジェクトが存在すること（通常の Issue へのコメントを除外するため）。
- **キーワード判定:** コメント本文（`payload.comment.body`）に **「レビューして」** という文字列が含まれていること。

### 3. コンテキスト（PR情報）の収集
Agent が精度の高いレビューを行うために、GitHub API（`@octokit/rest` など）を使用して以下の情報を取得します。
- **PR の概要:** PR のタイトルと Description（`issue.title`, `issue.body`）。
- **コメント履歴:** これまでに PR 上で交わされたコメントやレビューの履歴。
- **変更内容 (Diff):** PR で変更されたファイルの一覧と、具体的な差分（Diff）。

### 4. Agent (LLM) によるレビュー実行
収集した「PRの概要」「コメント履歴」「変更差分」をコンテキストとしてまとめ、LLM にプロンプトとして渡します。
LLM は **Gemini** をデフォルトで使用しますが、将来的に他のモデル（OpenAI, Anthropic 等）にも容易に切り替えられるよう、Vercel AI SDK のような抽象化レイヤーを挟むか、インターフェースを切った設計とします。

**プロンプト設計のイメージ:**
- 役割: あなたは優秀なシニアエンジニアです。
- タスク: 以下の PR 概要、既存のコメント、変更差分を読み、コードの品質、バグの可能性、可読性についてレビューしてください。
- 出力形式: GitHub の Markdown に適した形式で、改善提案や指摘事項を出力してください。

### 5. レビュー結果のフィードバック
LLM の生成結果を受け取り、再び GitHub API を使用して PR にコメントを投稿します。
- 全体へのコメントであれば `issues.createComment`。
- コード行に対する具体的な指摘が可能であれば、`pulls.createReview` を用いてフォーマルなレビューとして投稿します。

## システム上の考慮事項 (非同期処理とフィードバック)
GitHub Webhook は 10秒以内 にレスポンスを返す必要があります。API によるデータ収集や LLM の推論には時間がかかるため、Webhook エンドポイントはリクエストを受け付けた段階で直ちに `200 OK` を返し、実際のレビュー処理はバックグラウンドの非同期処理として実行します。

また、ユーザー体験向上のため、以下のフローで状態をフィードバックします：
1. **処理開始:** `200 OK` を返す前に（または直後に）、対象の PR に対して「👀 レビュー中...」といったプレースホルダーとなるコメントを投稿します。
2. **バックグラウンド処理:** コンテキストの収集と LLM によるレビューを実行します。
3. **結果の更新:** 処理が完了、あるいはエラーで失敗した際に、ステップ1で投稿したコメントの ID を指定して、内容を最終的なレビュー結果またはエラーメッセージに更新（Update）します。
